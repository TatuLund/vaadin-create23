# Copilot Workspace Instructions

This file is the compact Copilot entrypoint for the workspace.

The authoritative repository-specific instructions live in `AGENTS.md`. Keep detailed guidance there and keep this file short to avoid drift.

## Source of truth
- Read `AGENTS.md` first for repo structure, commands, test setup, architecture boundaries, Vaadin specifics, local environment details, and documentation references.

## Quick rules
- Tech stack: Java 21, Vaadin 8.31.1, GWT 2.13.0, Hibernate ORM, Redis event bus, JUnit 4, TestBench, JSoup, OpenTelemetry, SonarQube, JSpecify.
- Do not use Lombok.
- Do not use Spring DI or other CDI frameworks.
- Respect the application flow: `View -> Presenter -> Service -> DAO -> HibernateUtil -> Database`.
- Services are singletons accessed through static `get()` style methods.
- Use `@NullMarked` on public classes and `@Nullable` where null is allowed.
- Prefer records for DTOs, EventBus events, and other immutable data holders.
- Entity `equals()` and `hashCode()` must be based on `id`.
- Put CSS class name constants in `VaadinCreateTheme`.
- Keep utility methods static.
- Sanitize displayed user HTML with JSoup content mode.

## Testing expectations
- New business logic needs JUnit 4 tests.
- New view components need UI Unit Tests.
- New custom components with client-side code need TestBench integration tests.
- UI unit tests should follow the `AbstractUITest` + `mockVaadin(ui)` + `navigate(View.NAME, View.class)` pattern documented in `AGENTS.md`.
- Do not remove or weaken existing tests without confirming the behavior change is intentional.

## Workflow
- Run `mvn spotless:apply` when formatting is needed.
- Use `mvn verify -Pit` for the full verification path when a change affects integration-tested behavior.
- Check `AGENTS.md` for the preferred dev commands, mock-data caveats, and ArchUnit-enforced boundaries before making larger changes.
